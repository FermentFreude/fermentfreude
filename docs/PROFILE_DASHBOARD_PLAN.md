## 🔐 Part 3: Security & Access Control Architecture

### 3.1 Authentication Flow

```
User Registration:
  1. (Anonymous) POST /api/users/create with email/password/confirm
  2. Backend validates email uniqueness, password strength
  3. Payload creates User doc with hashed password
  4. Auth context sets user state + JWT token in cookies
  5. User is logged in automatically
  6. [FUTURE] Send verification email, delay account activation

Login Flow:
  1. (Anonymous) POST /api/users/login with email/password
  2. Backend validates credentials
  3. Payload returns JWT token
  4. Auth context sets user state + stores token in cookies
  5. /api/users/me endpoint called to populate user data
  6. Session persists across page reloads

Logout Flow:
  1. (Authenticated) POST /api/users/logout
  2. Backend clears JWT from cookies
  3. Auth context sets user to null

Session Check:
  1. On mount, useAuth hook calls /api/users/me
  2. If valid token: returns user data
  3. If invalid/expired: returns 401, sets user to null
```

### 3.2 Access Control Rules

**Users Collection:**

```typescript
// Anyone can create an account
create: publicAccess

// Users see their own profile, admins see all
read: adminOrSelf

// Users update their own profile, admins can update any
update: adminOrSelf

// Only admins can delete users
delete: adminOnly

// Only admins can access user dashboard
admin: checkRole(['admin'], user)
```

**Orders Collection (via ecommerce plugin):**

```typescript
read: (user) => {
  // Admins see all orders
  if (checkRole(['admin'], user)) return true
  // Customers see their own orders
  return {
    customer: { equals: user.id }
  }
}
```

**Addresses Collection (via ecommerce plugin):**

```typescript
read/create/update: (user) => {
  // Customers manage their own addresses
  // Admins see all
}
```

### 3.3 Form Validation Strategy

**Client-Side (React Hook Form + Zod):**
- Real-time validation
- User feedback with error messages
- Password strength indicator
- Email format validation
- Address field validation

**Server-Side (Payload + API routes):**
- Validate all inputs (never trust client)
- Check email uniqueness in DB
- Enforce minimum password requirements
- Rate limiting (configure in API routes)
- CSRF protection (Next.js built-in)

```typescript
// src/utilities/validators.ts
import { z } from 'zod'

export const passwordSchema = z
  .string()
  .min(8, 'Password must be at least 8 characters')
  .regex(/[A-Z]/, 'Password must contain an uppercase letter')
  .regex(/[a-z]/, 'Password must contain a lowercase letter')
  .regex(/[0-9]/, 'Password must contain a number')

export const addressSchema = z.object({
  street: z.string().min(5, 'Street address required'),
  city: z.string().min(2, 'City required'),
  state: z.string().optional(),
  postalCode: z.string().regex(/^[0-9]{5,}$/, 'Valid postal code required'),
  country: z.string().min(2, 'Country required'),
  type: z.enum(['home', 'work', 'other']),
  isDefaultShipping: z.boolean().default(false),
  isDefaultBilling: z.boolean().default(false),
})

export type AddressFormData = z.infer<typeof addressSchema>
```

    defaultValues: initialData,
  })

  return (

        <Button type="submit" disabled={isSubmitting} className="bg-ff-gold text-ff-charcoal">
          {isSubmitting ? 'Saving...' : 'Save Address'}
        </Button>
        <Button type="button" variant="outline" onClick={() => window.history.back()}>

      </div>
    </form>
  )
