02 Workshops — Struktur & Buchungsablauf

Workshops sind dreischichtig aufgebaut:

Schicht 1
Workshop
Das Thema (z. B. Grundkurs). Preis, Beschreibung, SEO.
Schicht 2
Termin
Datum + Uhrzeit, max. 12 Plätze, verfügbare Plätze, veröffentlicht ja/nein.
Schicht 3
Buchung
Name + E-Mail der Kund:in, zugehöriger Termin, Status.

Buchungsablauf (Workshop-Ticket)

Kund:in wählt Workshop + Datum im Shop aus

Kund:in bezahlt per Stripe (Kreditkarte / SEPA)

Stripe bestätigt Zahlung → System erstellt Bestellung

Automatisch: Buchung wird auf „confirmed" gesetzt, verfügbare Plätze werden um 1 reduziert

Automatisch: Bestätigungs-E-Mail mit Datum, Uhrzeit & Adresse wird gesendet

Bestellung erhält Status „Completed" (im Admin sichtbar)

Buchungsstatus Bedeutung Wie gesetzt?
pending Buchung erstellt, Zahlung noch ausstehend Automatisch bei Checkout
confirmed Zahlung eingegangen, Platz reserviert Automatisch nach Stripe-Bestätigung
cancelled Storniert (Platz wird wieder frei) Manuell im Admin
refunded Rückerstattet Manuell im Admin
Kapazität
Jeder Termin hat maximal 12 Plätze. Wenn alle Plätze belegt sind, sollte der Termin im Shop nicht mehr buchbar erscheinen. Wenn ihr einen Termin storniert → manuelle Anpassung der verfügbaren Plätze im Admin nötig.
03 Physische Produkte — Ablauf nach Zahlung

Für Eingemachtes, Frisches und Flaschen (jarred / fresh / bottled):

Kund:in kauft im Shop und bezahlt per Stripe

Zahlung geht ein → Bestellung erscheint im Admin unter Orders

Bestellbestätigungs-E-Mail geht automatisch an Kund:in raus

Lagerbestand wird automatisch um die bestellte Menge reduziert

Ihr handelt: Produkt vorbereiten, verpacken

Ihr handelt: Kund:in benachrichtigen (z. B. „Abholung bereit" oder Versand-Info) — derzeit manuell außerhalb des Systems

Bestellung bleibt auf Status „Completed" (wird sich mit Abschnitt 9 ändern)

Fehlende Funktion (bekannt)
Es gibt keine automatische Versand-E-Mail. Der Code dafür existiert, aber er überwacht einen Status („shipped"), der technisch nie gesetzt werden kann. Das ist ein bekannter Fehler, der behoben werden soll.
Workaround bis zur Lösung
Kund:innen bei physischen Bestellungen manuell per E-Mail informieren, wenn ihre Bestellung fertig ist. Oder eine kurze Brevo-E-Mail manuell auslösen.
04 Online-Kurse — Ablauf nach Zahlung

Kund:in kauft Online-Kurs und bezahlt per Stripe

Zahlung geht ein → System erstellt Bestellung

Bestellbestätigungs-E-Mail geht automatisch raus

Automatisch: Kurs wird für das Kund:innen-Konto freigeschaltet

Kund:in kann sich einloggen und sofort mit dem Kurs beginnen

Bestellung erhält Status „Completed"

Voll automatisch
Online-Kurse laufen komplett automatisch. Kein manueller Schritt nötig.
05 Gutscheine (Gift Vouchers)

Jemand kauft einen Gutschein im Shop (€ 99, fixer Preis)

Zahlung geht ein → System generiert automatisch einen einzigartigen Code (Format: FF-GIFT-XXXXXXXX)

Gutschein-E-Mail mit dem Code geht an Käufer:in raus

Beschenkter gibt Code beim Checkout ein → Betrag wird abgezogen

Gutschein wechselt von Status active auf redeemed

Gutscheinstatus Bedeutung
active Gültig, noch nicht eingelöst
redeemed Eingelöst und verbraucht
expired Abgelaufen (kann manuell im Admin gesetzt werden)
Gutschein-Verwaltung
Alle Gutscheine sind im Admin unter Vouchers einsehbar. Ihr könnt dort prüfen, ob ein Code eingelöst wurde und Gutscheine ggf. auf expired setzen.
06 Lagerbestand (Inventar)

Der Lagerbestand für physische Produkte wird automatisch verwaltet:

Aktion Was passiert mit dem Lager? Automatisch?
Bestellung erfolgreich Menge wird abgezogen Auto
Bestellung storniert / rückerstattet Menge wird wieder gutgeschrieben Auto
Ihr produziert mehr Menge manuell im Admin erhöhen Manuell
Lager = 0 Produkt sollte als „ausverkauft" erscheinen Auto
Wichtig
Workshop-Tickets und Online-Kurse haben keinen klassischen Lagerbestand. Workshops haben Plätze pro Termin (max. 12). Online-Kurse sind unbegrenzt verfügbar.

08 Bestellstatus — was bedeutet was?

Jede Bestellung hat einen von vier möglichen Status-Werten:

processing
→
completed
·
cancelled
·
refunded
Status Was bedeutet das? Wie wird es gesetzt?
processing Zahlung ausstehend oder in Bearbeitung Automatisch beim Erstellen der Bestellung
completed Zahlung eingegangen (derzeit: sofort nach Stripe-Bestätigung) Automatisch — wird sich mit Abschnitt 9 ändern
cancelled Bestellung storniert Manuell im Admin oder automatisch bei fehlgeschlagenem Stripe-Webhook
refunded Betrag zurückerstattet Manuell im Admin, nachdem Rückerstattung in Stripe veranlasst wurde
Das aktuelle Problem
Alle Bestellungen — auch physische Produkte — springen sofort nach Zahlung auf „completed". Das macht es schwer zu erkennen, welche Bestellungen noch bearbeitet werden müssen. Abschnitt 9 zeigt die geplante Lösung.
09 Geplant: Fulfillment-Lifecycle & offene Fragen

Um physische Bestellungen besser zu verwalten, ist ein neuer Fulfillment-Status geplant — zusätzlich zum bestehenden Bestellstatus. Dieser würde euch zeigen, in welchem Stadium der Bearbeitung eine Bestellung ist.

Geplante Fulfillment-Status

Status Bedeutung Wer setzt ihn?
pending Zahlung eingegangen — noch nichts vorbereitet Automatisch
preparing Ihr bereitet die Bestellung vor Manuell im Admin
ready_for_pickup Fertig — Kund:in kann abholen ODER wird versendet Manuell im Admin → E-Mail geht automatisch raus
fulfilled Abgeholt oder versendet — abgeschlossen Manuell im Admin

Dieser Plan ist technisch fertig beschrieben. Bevor er umgesetzt wird, brauchen wir eure Antworten auf drei Fragen:

Drei Fragen für David & Marcel

1
Wie kommt eure Ware zu den Kund:innen?

Das bestimmt, welche Status-Bezeichnungen und E-Mail-Texte sinnvoll sind.

Nur Abholung vor Ort Nur Versand (Post/DHL) Beides je nach Produkt
2
Wollt ihr Bestellungen im Admin als „erledigt" abhaken können?

Oder reicht es, wenn Kund:innen automatisch eine E-Mail bekommen, sobald ihre Bestellung fertig ist?

Ja, Admin-Checkbox reicht Nein, nur die E-Mail Beides wäre ideal
3

Wenn ihr eine Bestellung auf „Abholbereit" setzt — was soll die E-Mail beinhalten? Abholadresse, Öffnungszeiten, Hinweise?
